222 A practical implementation stack for this is: • Python for orchestration and math glue. • FastAPI for the internal control plane and typed APIs. • vLLM to serve the local proposer model behind an OpenAI-compatible endpoint with structured outputs enforced by JSON schema, regex, or CFG grammar. • Qwen2.5-Coder-7B as the first proposer model: its model card lists it as a 7.61B code model, with a 131,072-token context length, though the shipped config is set to 32,768 unless you enable YaRN-style rope scaling for longer inputs. • SymPy for symbolic manipulation, Z3 for SMT checks, SageMath for combinatorics/enumeration, polymake for polyhedral geometry, and optionally Lean 4 for the small subset of results worth formalizing. FastAPI is standards-based and production-oriented; vLLM’s docs describe both OpenAI-compatible serving and structured-output backends; Z3’s official guide exposes Python usage; Sage’s combinatorics docs emphasize enumeration/listing/random generation; polymake is documented as open-source software for research in polyhedral geometry; Lean’s current theorem-proving text assumes Lean 4.26.0. Below is the implementation I would actually build.   ⸻   1. System topology Do not build this as one monolith. Build it as six services plus one shared schema package. Services 1. kb-service Owns Layer 0 records, retrieval, deduplication, provenance, and trust metadata. 2. generator-service Owns exact computation, enumeration, perturbation generation, and serialization of computed artifacts. 3. proposer-service Wraps the local 7B model. It never returns free text to the rest of the system. It only emits DSL-constrained objects. 4. checker-service Runs symbolic, numeric, combinatorial, and SMT checks. This is the scientific truth surface. 5. controller-service Owns search state, proposal queues, budgets, novelty scoring, and experiment scheduling. 6. review-service Produces analyst-facing summaries, substrate schemas, audit dashboards, and “what changed this week” reports. Shared package 7. pgcore Pure Python package with: • type definitions • schema validators • DSL parsers • canonical hashing • common feature extraction • stable IDs • provenance utilities Everything should import types from pgcore and never invent ad hoc JSON blobs.   ⸻   2. Repo layout A workable repo shape: pg-pipeline/ pyproject.toml README.md pgcore/ ids.py schemas/ base.py geometry.py conjecture.py check_result.py substrate.py dsl/ recurrence.py equivalence.py boundary.py primitive_set.py canon/ normalize.py hashing.py features/ boundaries.py residues.py symmetry.py scaling.py services/ kb/ app.py retrieval.py ingestion.py dedupe.py trust.py generator/ app.py families/ grassmannian.py cosmohedron.py surfaceology.py perturb.py serializers.py proposer/ app.py prompts/ normalize.txt recurrence.txt equivalence.txt repair.txt decoders.py adapters.py retrieval.py checker/ app.py symbolic.py numeric.py combinatorial.py smt.py formal.py controller/ app.py scheduler.py bandit.py novelty.py budgets.py experiments.py review/ app.py weekly.py substrate_reports.py audit.py configs/ models/ proposer-qwen7b.yaml families/ grassmannian_small.yaml cosmohedron_scalar.yaml checks/ recurrence.yaml equivalence.yaml data/ kb/ generated/ checks/ reports/ tests/ unit/ integration/ golden/   ⸻   3. Data model: the most important implementation detail The pipeline will live or die on schema discipline. You need immutable records plus derived indices. Never mutate canonical records in place except to append new provenance or correction metadata. 3.1 Base record contract Every stored object should have this envelope: { "id": "geom:grassmannian_cell:sha256:...", "kind": "geometry_record", "family": "positive_grassmannian_cell", "version": "1.0", "created_at": "2026-04-19T18:25:00Z", "producer": { "type": "generator|human|checker|import", "name": "generator-service", "version": "0.3.1" }, "provenance": { "source_type": "literature|exact_computation|numeric_only|conjectural", "citation_keys": ["paper:..."], "input_artifact_ids": ["..."], "generator_run_id": "run_...", "notes": "" }, "trust": { "level": "high|medium|low", "reason": "exact symbolic computation" }, "scope": { "parameters": {"n": 6, "k": 2, "m": 4}, "representation": "boundary_lattice_v1", "applicability_domain": "tree-level only" }, "payload": {} } 3.2 Record kinds you actually need You need at least these first-class types: • geometry_record • representation_map • invariant_record • failed_conjecture_record • proposal_record • check_result_record • stress_result_record • substrate_schema_record • review_decision_record If you do not separate these, the search controller will end up treating verified facts and model guesses as the same thing.   ⸻   4. Canonical IDs and deduplication Deduplication is not “same JSON string.” It is same mathematical object under canonicalization. 4.1 Canonical hashing strategy For each family, define: • a canonical ordering of indices • a canonical orientation convention • a canonical string serialization • optional quotienting under known symmetries Then hash that canonical form. Example sketch: def canonical_id_for_boundary_lattice(nodes, edges, symmetry_group=None): norm_nodes = sorted(normalize_node(n) for n in nodes) norm_edges = sorted(tuple(sorted(map(normalize_node, e))) for e in edges) rep = {"nodes": norm_nodes, "edges": norm_edges} if symmetry_group: reps = [apply_symmetry(rep, g) for g in symmetry_group] rep = min(reps, key=stable_serialize) return "geom:boundary_lattice:sha256:" + sha256(stable_serialize(rep).encode()).hexdigest() 4.2 Why this matters Without canonical IDs: • the proposer will rediscover the same object under relabeling, • the search controller will overcount novelty, • the checker cache will miss identical cases, • and your KB will rot.   ⸻   5. Layer 0 implementation: structured domain memory This should not start as a vector DB. Start as typed records + exact filters + derived embeddings only for convenience. 5.1 Storage choice Use: • relational store for metadata and search state • filesystem or object store for larger artifacts • precomputed materialized views for retrieval For a solo build, this can be: • SQLite or Postgres for metadata • Parquet/JSONL for bulk generated artifacts • local filesystem for blobs 5.2 Retrieval API Retrieval should have three modes: Exact retrieval GET /kb/records?family=positive_grassmannian_cell&n=6&k=2&representation=sign_flip Constraint retrieval POST /kb/query { "family": "cosmohedron_face", "filters": { "trust.level": "high", "scope.parameters.order": {"$lte": 7} }, "sort": ["scope.parameters.order", "created_at"] } Similarity retrieval Only after exact retrieval fails. Similarity should run over structured features, not raw prose. 5.3 Ingestion pipeline Every paper-derived or hand-entered example goes through: 1. normalize fields 2. canonicalize representation 3. assign stable ID 4. validate against schema 5. attach provenance 6. compute derived features 7. store immutable record 8. update family-level indices 5.4 Feature views Precompute family-specific feature vectors, for example: { "feature_vector": { "num_faces": 42, "max_codim": 5, "boundary_degree_histogram": [1, 4, 9, 6], "sign_flip_signature": [0, 1, 1, 0, 2], "symmetry_rank": 3, "residue_support_size": 12 } } These are for retrieval and novelty, not as a substitute for exact structure.   ⸻   6. Layer 1 implementation: generator-service This service should be deterministic, resumable, and family-specific. 6.1 Job contract { "job_id": "gen_000123", "family": "positive_grassmannian_cell", "task": "enumerate_cells", "params": {"n": 7, "k": 2, "max_dim": 4}, "compute_mode": "exact", "seed": 17, "artifact_targets": ["boundary_lattice", "cell_count", "sign_flip_encoding"] } 6.2 Generator workers Implement separate worker classes per family: class FamilyGenerator(Protocol): def run(self, params: dict) -> list[GeometryRecord]: ... def perturb(self, rec: GeometryRecord, mode: str) -> list[GeometryRecord]: ... def family_checks(self) -> list[str]: ... Then: class GrassmannianGenerator(FamilyGenerator): ... class CosmohedronGenerator(FamilyGenerator): ... class SurfaceologyGenerator(FamilyGenerator): ... 6.3 Mutation generation Mutation is not optional. Build it into Layer 1 from day one. Mutation types: • label permutation • representation-preserving reorder • minor parameter perturbation • structure-breaking deletion • synthetic near-miss object generation Example: def generate_mutations(record): return [ relabel(record, random_perm(record.labels)), reorder_equivalent_facets(record), drop_nonessential_annotation(record), break_required_adjacency(record), create_near_miss(record) ] 6.4 Artifact storage Store generated outputs in two forms: • compact record for DB • raw full artifact for later audit Example: data/generated/positive_grassmannian_cell/2026-04-19/run_123/ manifest.json records.jsonl raw/ cell_001_boundary_lattice.json cell_001_signflip.txt 6.5 Determinism For every generator run, log: • code version • dependency versions • seed • numeric precision • time budget • fallback behavior If a conjecture later depends on a computation, you need reproducibility without archaeology.   ⸻   7. Layer 2A implementation: normalization/compression This stage should be deterministic where possible. Only use the model when canonicalization is hard and the output can be tightly validated. 7.1 When not to use the model Do not ask the model to normalize: • sorted incidence lists • index canonicalization • exact symmetry quotienting • simple basis conversions you can code Write exact code for those. 7.2 When to use the model Use the model only for: • mapping two partially different encodings into the same intermediate schema, • proposing feature tags from structured objects, • recovering a missing but checkable normalization field. 7.3 Intermediate normalized schema Every geometry should get mapped into a narrow common form: { "normalized_object": { "family": "positive_grassmannian_cell", "params": {"n": 6, "k": 2}, "canonical_representation": "boundary_lattice_v1", "symmetry_signature": { "automorphism_count": 4, "orbits": [[0,1],[2,3,4]] }, "feature_summary": { "face_count": 14, "codim_histogram": [1,4,6,3], "adjacency_spectrum": [2,2,3,3,4] }, "derived_labels": [ "connected_boundary", "sparse_residue_support" ] } } 7.4 Validation Validation here should reject: • missing required keys • impossible symmetry claims • derived labels inconsistent with exact features • family mismatch   ⸻   8. Layer 2B implementation: proposer-service This is the small model box. Keep it brutally narrow. 8.1 Model serving Serve the model locally through vLLM. Use: • OpenAI-compatible API • structured outputs • low temperature for repair/classification • moderate temperature for novelty search • fixed max tokens per task type vLLM’s docs explicitly support structured outputs through JSON schema, regex, and CFG grammar, both in OpenAI-compatible serving and offline inference. 8.2 Task-specific DSLs Do not use one giant JSON schema. Use one DSL per task. Recurrence DSL { "task": "recurrence", "family": "cell_count_sequence", "scope": {"n_range": [4, 10], "k": 2}, "rule": { "type": "linear_recurrence", "order": 2, "coefficients": ["n-1", "-2"], "offset_terms": ["a[n-1]", "a[n-2]"] }, "confidence_hint": 0.41 } Equivalence DSL { "task": "equivalence", "lhs_id": "geom:...", "rhs_id": "geom:...", "map_type": "representation_change", "mapping": { "basis_permutation": [2,0,1,3], "sign_rule": "preserve_orientation" } } Primitive-set DSL { "task": "primitive_set", "family": "transition_family_alpha", "primitives": [ {"name": "boundary_take", "arity": 1}, {"name": "glue", "arity": 2}, {"name": "residue_extract", "arity": 1} ], "closure_claim": "all_observed_transitions_generated" } 8.3 Prompt template structure Each proposer prompt should have exactly four sections: 1. Role and hard constraints 2. Retrieved structured examples 3. Task schema 4. Reject conditions Example shape: You are emitting ONLY a JSON object conforming to the attached schema. Do not explain. If no good candidate exists, emit {"task":"recurrence","status":"no_candidate"}. Known examples: <json examples> Known failures: <json failures> Target: Propose one recurrence that matches these observations and differs materially from known failures. Schema: <json schema> 8.4 Retry policy Retries should not be generic. Make them diagnostic. • parse failure → retry with parser error • schema failure → retry with validation diff • duplicate of known candidate → retry with novelty constraint • repeated bad family scope → reduce context and sharpen examples 8.5 Fine-tuning You do not need to fine-tune first. Start with prompt-only + retrieval + grammar constraints. Fine-tune only after you have: • at least 500–1000 good input/output pairs • at least 100 hard negative examples • clear task segmentation by DSL Training examples should be pairs like: • normalized inputs • correct recurrence/equivalence/repair object • plus optional checker feedback for failed proposals Do not train on prose reasoning. 8.6 Decoding parameters Use different decoding profiles: • repair/profile temperature=0.1, top_p=0.9 • classification/profile temperature=0.0 • novel conjecture/profile temperature=0.5, top_p=0.95, multiple samples • fallback/profile lower temperature and narrower schema   ⸻   9. Layer 3 implementation: checker-service This is the hardest engineering layer after Layer 0. 9.1 Unified checker interface All checkers should return the same envelope: { "proposal_id": "prop_...", "check_type": "symbolic|numeric|combinatorial|smt|formal", "status": "pass|fail|inconclusive", "evidence": { "tested_cases": ["geom:...", "geom:..."], "counterexamples": [], "metrics": {"max_abs_err": 0.0} }, "failure_trace": null, "cost": {"seconds": 2.1, "memory_mb": 184}, "checker_version": "0.2.0" } 9.2 Symbolic checker Use SymPy or external CAS wrappers when needed. Responsibilities: • substitute proposed recurrence into exact computed values • simplify identities • check basis transforms • verify residue equalities where symbolic form exists Implementation sketch: def check_linear_recurrence(rule, samples): for s in samples: lhs = sympify(rule.lhs).subs(s) rhs = sympify(rule.rhs).subs(s) if simplify(lhs - rhs) != 0: return fail(counterexample=s) return pass_() 9.3 Combinatorial checker Responsibilities: • boundary incidence consistency • face-count consistency • closure of primitive sets on observed transitions • automorphism and orbit sanity This checker should not depend on the proposer’s representation. Convert both proposal and target into a family-native intermediate form first. 9.4 Numeric checker Use arbitrary precision where identities are only numerically checkable. Responsibilities: • high-precision evaluation of candidate formulae • perturbation sensitivity • agreement thresholds • independent numeric verification path Important: numeric checks should store precision settings and tolerances explicitly. 9.5 SMT checker Use Z3 for: • consistency of structural constraints • existence of a mapping under finite combinatorial conditions • equivalence of constrained encodings • bounded obstruction search Example: from z3 import * def check_boundary_map(lhs, rhs): s = Solver() perm = [Int(f"p{i}") for i in range(len(lhs.nodes))] # bijection constraints s.add(Distinct(*perm)) for p in perm: s.add(p >= 0, p < len(rhs.nodes)) # adjacency preservation for i, j in lhs.edges: s.add(Or([And(perm[i] == a, perm[j] == b) for a, b in rhs.edges])) return s.check() 9.6 Formal checker Use Lean only for: • stable mini-lemmas • reusable proof kernels • finite certified cases • algebraic identities you repeatedly rely on Do not route every candidate into Lean. That will stall the system. 9.7 Multi-view policy A proposal only advances if: • it passes at least two distinct check families, or • one check family is exact and the other is a strong held-out stress result Anything less is “interesting,” not “surviving.”   ⸻   10. Layer 4 implementation: search controller This is not just a queue. It is the part that prevents the pipeline from becoming a toy. 10.1 State tracked per proposal family For each conjecture family, track: • count attempted • count unique • pass rates by checker type • novelty distribution • mutation survival rate • held-out survival rate • analyst-reviewed substrate relevance • last-seen timestamp 10.2 Scoring A good first scoring function: score = ( 2.0 * multi_view_pass_rate + 1.5 * heldout_survival + 1.2 * mutation_survival + 1.0 * novelty_score + 1.5 * substrate_prior - 1.2 * similarity_to_failed - 0.8 * compute_cost_norm ) 10.3 Novelty score Novelty should be measured structurally: • tree edit distance on recurrence ASTs • graph similarity on boundary mappings • primitive-set overlap coefficients • feature-based clustering distance Not string distance. 10.4 Budgeting Have separate budgets: • exploration budget • exploitation budget • repair budget • mutation budget • analyst-triggered budget Otherwise repair loops will eat everything. 10.5 Scheduler Simple but effective: • top-level family priority queue • within family: Thompson sampling or UCB-style allocation • reserve 10–20% for random exploration That is enough. You do not need a fancy agent planner.   ⸻   11. Layer 5 implementation: stress harness This is where fake patterns die. 11.1 Stress result schema { "proposal_id": "prop_...", "suite": "heldout_large_n|symmetry_preserve|symmetry_break|near_miss", "cases_tested": 24, "passes": 21, "fails": 3, "failure_examples": ["geom:..."], "robustness_score": 0.81 } 11.2 Suites to implement first Start with four suites only: 1. larger n / order 2. same object, different encoding 3. one essential structure broken 4. near miss generated from nearest-neighbor examples 11.3 Near-miss generation This matters more than people think. For each accepted-looking object, generate: • one case with preserved surface statistics but broken key adjacency • one case with same counts but changed incidence • one case with shuffled labels that should be equivalent • one case with perturbed residues that should fail exact equality If the conjecture cannot separate these, it is not substrate-grade.   ⸻   12. Layer 6 implementation: substrate translation schema This should be a typed form, not a dashboard paragraph. 12.1 Schema { "proposal_id": "prop_...", "object_class": "positive_grassmannian_transition", "invariant_or_law": "boundary_adjacency_preservation", "preservation_domain": {"family": "alpha", "params": {"n_max": 10}}, "failure_domain": {"mutations": ["edge_drop", "orientation_flip"]}, "evidence_type": ["symbolic", "combinatorial", "heldout"], "scaling_behavior": { "classical_description_cost": "O(n^3)", "geometric_state_cost": "empirically_flat_for_tested_range" }, "primitive_operations_implied": [ "boundary_take", "glue" ], "representation_dependence": "low", "readout_implication": "reconstructing full microstate destroys apparent gain", "resource_implication": "native adjacency preservation required", "candidate_substrate_constraint": "support adjacency-preserving transition primitive natively", "confidence_level": "medium", "open_ambiguity": [ "not yet tested outside family alpha" ] } 12.2 Human workflow You review these records, not raw checker logs. The review UI should let you: • compare surviving schema records side by side • approve / reject / defer • attach notes • raise required new checks • promote constraints into a cumulative substrate spec   ⸻   13. Review-service and analyst UX You need three views. 13.1 Survivor board Shows: • proposal family • current score • checker passes • stress passes • novelty • substrate prior • analyst status 13.2 Failure board Shows: • top recurring failure modes • checker blind spots • high-volume duplicate proposals • normalization failures • schema drift 13.3 Substrate board Shows accepted constraints grouped by: • native state • primitive operations • readout • scaling • noise tolerance This is the one that matters to your actual goal.   ⸻   14. APIs between services Use small, typed internal APIs. 14.1 proposer-service POST /propose/recurrence POST /propose/equivalence POST /propose/repair POST /normalize/object 14.2 checker-service POST /check/symbolic POST /check/combinatorial POST /check/numeric POST /check/smt POST /check/formal 14.3 controller-service POST /jobs/enqueue POST /jobs/run-next POST /proposals/score POST /stress/run GET /dashboard/survivors Every API should accept and return schema-validated JSON only.   ⸻   15. Orchestration and execution loop This should be asynchronous but simple. 15.1 Main loop while True: task = controller.next_task() if task.type == "generate": records = generator.run(task.spec) kb.store(records) controller.update_generation(records) elif task.type == "normalize": normalized = proposer.normalize(task.records) kb.store(normalized) controller.update_normalization(normalized) elif task.type == "propose": proposal = proposer.propose(task.payload) kb.store([proposal]) controller.enqueue_checks(proposal) elif task.type == "check": result = checker.run(task.proposal, task.check_type) kb.store([result]) controller.update_from_check(result) elif task.type == "stress": stress = stress_harness.run(task.proposal) kb.store([stress]) controller.update_from_stress(stress) elif task.type == "review_refresh": review.build_views() 15.2 Queue semantics Use at-least-once execution with idempotent handlers. Every task should be safe to retry.   ⸻   16. Logging and observability You need scientific logs, not just server logs. 16.1 Per-run metadata For every proposal/check/stress run, log: • input record IDs • schema version • model version • prompt template version • decoding profile • checker version • compute cost • wall time • seed 16.2 Metrics Track: • parse success rate • schema-valid success rate • duplicate proposal rate • per-check pass rate • mutation fail rate • held-out fail rate • analyst acceptance rate • time from generation to substrate schema If analyst acceptance stays near zero, the loop is academic noise.   ⸻   17. Testing strategy You need four kinds of tests. 17.1 Unit tests For: • parsers • canonicalizers • hashing • schema validators • feature extractors 17.2 Golden tests Keep a small set of known objects and expected normalized/check outputs. These catch silent drift. 17.3 Adversarial tests Feed malformed but plausible objects: • wrong symmetry claims • duplicated edges • equivalent objects with relabeled nodes • recurrence with hidden undefined terms 17.4 End-to-end tests One narrow family, one generator, one proposer task, two checkers, one stress suite. You want this runnable in CI.   ⸻   18. MVP implementation order Build in this order. Week 1 • pgcore schemas • canonical IDs • KB ingestion • one family generator • one symbolic checker Week 2 • proposer service with strict JSON output • recurrence DSL • controller queue • first dashboard Week 3 • combinatorial checker • mutation generation • held-out stress suite • failure board Week 4 • substrate schema records • weekly review report • novelty scoring • first analyst loop Do not touch Lean until you already have proposals worth certifying.   ⸻   19. Exact places people usually get this wrong Three implementation mistakes will quietly kill this system. 19.1 They let the proposer emit prose Once you do that, every downstream component becomes a brittle parser. 19.2 They treat retrieval as “semantic search over papers” Wrong. Retrieval should be over typed examples, failures, and exact artifacts. 19.3 They do not encode known failures A pipeline without failed conjectures in the KB is a rediscovery machine.   ⸻   20. A minimal concrete slice If I were building the first vertical slice, I would choose: • family: small positive-Grassmannian cell counts or boundary lattices • task: recurrence discovery • proposer output: recurrence DSL only • checks: symbolic + combinatorial • stress: larger n + one broken-structure mutation • analyst output: one substrate schema field, not ten Example vertical slice flow 1. generator enumerates cases for n=4..8, k=2 2. KB stores counts and normalized boundary summaries 3. proposer gets 6–8 examples + known failures 4. proposer emits 20 recurrence candidates under grammar 5. checker kills 17 immediately 6. 3 survive symbolic checks 7. stress harness breaks 2 under mutation 8. 1 survives held-out n=9,10 9. review layer writes: • invariant/law • tested domain • failure domain • candidate substrate implication That is the first useful loop.   ⸻   21. Where the real bottlenecks will be Not inference speed. The bottlenecks will be: • canonicalization quality • checker coverage • mutation design • and whether your normalized representations preserve the real structure That is why I would spend more engineering time on pgcore/canon, checker/, and generator/perturb.py than on model tuning.   ⸻   22. My recommended defaults If you want concrete defaults rather than options: • use FastAPI for service wrappers • use vLLM for local inference and grammar-constrained output • start with Qwen2.5-Coder-7B • use SymPy + Z3 first • add Sage/polymake when the family needs them • postpone Lean • store records as immutable JSONL + metadata DB • treat the KB as the central artifact • make every proposal family-specific and DSL-specific • log everything needed for replay That is the implementation shape I would trust. The next useful step is to turn this into a concrete interface spec: actual Pydantic-style schemas for GeometryRecord, ProposalRecord, CheckResultRecord, and SubstrateSchemaRecord, plus one end-to-end recurrence example.

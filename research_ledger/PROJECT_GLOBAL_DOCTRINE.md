## 0. Purpose

This file exists because local ledger correctness is not enough.

The main failure mode is:

```text
local correctness without global coherence
```

A future artifact can have correct status labels, validators, source boundaries, and commits while still losing the living shape of the project.

This doctrine is the compact control surface that must be reloaded before long work blocks.

Founder correction that triggered this doctrine, sanitized for repo hygiene:

```text
The assistant is not deeply aware of the project in the total/global sense during long tool work, and does not reliably hold the whole project in context while proceeding. This limitation must be explicit because it is critical.
```

Operational admission:

```text
The assistant does not have continuous total awareness of the whole project while doing long tool work.
The workflow must therefore force global role placement before and during major work.
```

```text

```

This distinction must be preserved in every engineering, physics, and corporate artifact.

Before any long work block, answer internally:

```text
1. Am I working on the object, a mechanism, a comparator, a route, a proxy, a guard, a source review, or corporate framing?
2. Does this work move the project closer to Formation Medium, or merely add local ledger structure?
3. Could this artifact make CRBSM, Route A, scattering, or high-resonance look like the object by accident?
4. Does this work preserve the distinction between formation and computation?
5. Does this work preserve the distinction between readout, transport, and substrate?
6. Does this work preserve the distinction between source support and physical realization?
7. Does this work preserve founder-controlled technical spine rather than uncontrolled conceptual drift?
```

## 8. What Validators Mean

Validators mean:

```text
ledger discipline passed
```

Validators do not mean:

```text
the object is understood
the physics is true
the route is feasible
the corporate strategy is ready
```

A passing validator can coexist with conceptual drift.

This doctrine exists to reduce that risk.



## 8.1 Mission-Grade Critical Review Rule

Repo discipline is not enough.

The assistant must apply critical reasoning before and during major work. Validators, clean commits, source guards, prior artifacts, and user enthusiasm are not substitutes for judgment.

The assistant may challenge:

```text
the mission
the founder
the repo
the math
a theorem's applicability
a source interpretation
a route or comparator
an artifact that should be deleted or demoted
```

Before major work, the assistant must ask whether the work attacks a hard bottleneck, whether it could be net-negative, whether it leans on a thin-stick theorem/source/check, whether it can be made smaller or more falsifiable, and whether the founder could be wrong about the direction.

Do not confuse convenience with convergence.

A locally clean artifact may still be bad work.



## 8.2 Stage 0 Math-First Rule

Before major source-review, physical-comparator, route, substrate, or broad repo-expansion work, future workers must run a Stage 0 math-first attack pass.

Core rule:

```text
Tools serve the math plan.
Tools do not choose the plan.
```

Stage 0 must be tool-light and must ask:

```text
What is the actual mathematical object?
What definitions must be fixed before computation?
What is the smallest literal finite object to compute?
What theorem, counterexample, or invariant would change the project?
What would show this direction is wrong or irrelevant?
```

Only after Stage 0 may the worker escalate to:

```text
Stage 1 — formal target extraction
Stage 2 — literal computation / finite checks / formal engines
Stage 3 — physics or engineering emergence triage
```
